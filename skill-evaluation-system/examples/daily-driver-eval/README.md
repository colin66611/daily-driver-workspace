### 第三步: 评估结果分析

#### 使用评估标准:
```bash
# 查看评分标准
cat daily-driver-eval-workspace/evaluation-criteria.md
```

#### 计算总分:
```
总分 = 功能性 × 0.40 
     + 流程完整性 × 0.25 
     + 文件可靠性 × 0.20 
     + 边界处理 × 0.10 
     + 触发准确性 × 0.05
```

#### 判断评级:
- **90-100分**: 生产级 - 可直接使用
- **75-89分**: 可用级 - 基本可用,改进已知问题
- **60-74分**: 测试级 - 需要测试完善,修复核心问题
- **0-59分**: 开发级 - 不可用,重新开发核心功能

## 📊 核心评估指标

### 必须通过的检查点 (Critical):
1. ✅ 文件创建: task_plan.md, findings.md, progress.md 正确创建
2. ✅ 文件内容: 三文件包含所有必需字段
3. ✅ 状态更新: 任务完成时状态正确更新
4. ✅ 流程闭环: 从启动到收尾完整闭环
5. ✅ 不崩溃: 任何场景都不崩溃

### 重要检查点 (Important):
1. ⚠️ 任务延续: 昨天未完成任务正确检测和延续
2. ⚠️ 实时更新: 执行过程中文件实时更新
3. ⚠️ 并行执行: 多任务并行管理
4. ⚠️ 用户指引: 每阶段用户指引清晰
5. ⚠️ 错误提示: 异常场景友好提示

## 🆘 常见问题

### Q1: 是否需要执行所有测试用例?
**A**: 不需要。优先执行高优先级测试,确保核心功能正常。中低优先级可根据时间和需求选择性执行。

### Q2: 如何判断测试是否通过?
**A**: 参考 `evaluation-criteria.md` 的评分标准,每个测试用例都有明确的验证点和通过条件。

### Q3: 测试失败怎么办?
**A**: 记录失败原因,分析根本原因,提出改进建议。可迭代改进后重新测试。

### Q4: 评估数据如何保存?
**A**: 建议在 `daily-driver-eval-workspace/test-results/` 目录下保存每次测试的结果和报告。
